# 0041 — Habit Performance Metrics API

## Problem

There's no way to see which habits are most/least completed, how completion rates trend over time, or compare habit performance. The existing Streaks page shows current/longest streaks and weekly counts, but lacks deeper analytics like completion rate (%), rankings, and trend data.

## Solution

Add a new analytics service and API endpoints that expose three core metrics:

1. **Completion rate per habit** — % of available days each habit was completed, excluding exempt weekdays and allowed skip days from the denominator
2. **Habit ranking** — habits sorted by completion rate (best to worst)
3. **Trends over time** — daily completion data + pre-aggregated weekly summaries

API-only scope. Frontend (new Analytics page) deferred to a separate task.

Supports both fixed periods (7d, 30d, 90d) and custom date ranges.

## Files to Change

### New: `src/services/analytics_service.py`

Analytics service with these methods:

#### `get_habit_completion_rates(user_id, start_date, end_date) -> list[HabitCompletionRate]`

For each active habit belonging to the user:

1. Query all `HabitLog` records in `[start_date, end_date]` for that habit (by `last_completed_date`)
2. Calculate **available days** (denominator):
   - Total calendar days in range
   - Subtract days matching `exempt_weekdays` (stored as comma-separated weekday numbers on Habit)
   - Subtract `allowed_skip_days` proportionally (if habit allows N skip days per week, subtract `N * weeks_in_range`)
   - Clamp denominator to minimum of 1
   - If habit was created after `start_date`, use `habit.created_at.date()` as effective start
3. Calculate **completed days** (numerator): count of distinct `last_completed_date` values in range
4. Rate = `completed_days / available_days` (capped at 1.0)

Return list sorted by completion rate descending (ranking).

#### `get_habit_trends(user_id, habit_id=None, start_date, end_date) -> HabitTrendData`

1. Query `HabitLog` records in range, optionally filtered by `habit_id`
2. **Daily data**: group by `last_completed_date`, count completions per day
3. **Weekly summaries**: group daily data into ISO weeks, calculate:
   - `week_start` (Monday date)
   - `completions` (count)
   - `available_days` (same logic as completion rate, for that week)
   - `rate` (completions / available_days)
4. If `habit_id` is None, aggregate across all active habits

#### `get_habit_rankings(user_id, start_date, end_date) -> list[HabitRanking]`

Wrapper around `get_habit_completion_rates` that enriches each entry with:
- `rank` (1-based position)
- `total_completions` (absolute count in range)
- `current_streak` (from existing streak service)
- `longest_streak_in_range` (max `streak_count` from logs in range)

### Modify: `src/core/repositories.py` — `HabitLogRepository`

Add one new method:

#### `get_completion_counts_by_date(user_id, start_date, end_date, habit_id=None) -> list[dict]`

Django ORM query:
- Filter `HabitLog` by user, date range, optional habit_id
- `.values('last_completed_date')` + `.annotate(count=Count('id'))`
- Returns `[{last_completed_date: date, count: int}, ...]`

This avoids loading all log objects into memory — single aggregation query.

Query params:
- `period` / `start_date` + `end_date` (same as above)
- `habit_id`: int (optional — omit for aggregate across all habits)

Returns: `HabitTrendData` with daily + weekly arrays.

## Algorithm: Available Days Calculation

```
function calculate_available_days(habit, start_date, end_date):
  # Adjust start if habit was created after range start
  effective_start = max(start_date, habit.created_at.date())
  if effective_start > end_date:
    return 0

  total_days = (end_date - effective_start).days + 1  # inclusive

  # Subtract exempt weekdays
  exempt_days = parse_exempt_weekdays(habit.exempt_weekdays)  # e.g. [5, 6] for Sat/Sun
  exempt_count = 0
  for day_offset in range(total_days):
    d = effective_start + timedelta(days=day_offset)
    if d.weekday() in exempt_days:
      exempt_count += 1

  available = total_days - exempt_count

  # Subtract allowed skip days proportionally
  if habit.allowed_skip_days > 0:
    weeks_in_range = total_days / 7
    skip_allowance = int(habit.allowed_skip_days * weeks_in_range)
    available = available - skip_allowance

  return max(available, 1)  # never zero
```

## Algorithm: Period Resolution

```
function resolve_date_range(period, start_date, end_date, user_timezone):
  today = get_user_today(user_timezone)

  if start_date and end_date:
    return (start_date, end_date)

  match period:
    "7d":  return (today - 6 days, today)
    "30d": return (today - 29 days, today)
    "90d": return (today - 89 days, today)
    default: return (today - 29 days, today)
```

## Edge Cases

- **New habits**: If habit was created 3 days ago but user requests 30d range, only count those 3 days as available.
- **No logs in range**: Return completion rate of 0.0, not an error.
- **Backdated completions**: Use `last_completed_date` (not `timestamp`) so backdated entries count for the correct day.
- **Deactivated habits**: Only include active habits in results. Inactive habits are excluded from rankings.
- **Duplicate logs same day**: `get_completion_counts_by_date` groups by date, so multiple logs on same day count as 1 completed day.
- **Custom range validation**: `start_date` must be <= `end_date`, max range 365 days. Return 400 if invalid.
- **Skip days edge case**: If allowed_skip_days * weeks > non-exempt days, clamp available to 1.
