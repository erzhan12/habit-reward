# Feature 0038: Real-time Dashboard Sync via WebSocket

## Description

When a habit is completed via the Telegram bot (or REST API), the web dashboard should update automatically without manual page refresh. Uses WebSocket to push a lightweight "refresh" signal to connected browsers, which then call Inertia's `router.reload()` to re-fetch dashboard data.

## Architecture

```
Telegram webhook → bot handler → habit_service.process_habit_completion()
                                            ↓
                                 connection_manager.notify_user(user_id)
                                            ↓
                                 WebSocket → Browser → router.reload()
```

**Technology choice:** FastAPI WebSocket with in-memory connection manager (no Redis, no Django Channels). Rationale:
- Bot and web server run in the same process (single Docker container)
- 1-5 concurrent users, SQLite database
- FastAPI already integrated via `CombinedASGIApp` in `asgi.py`
- Zero new Python dependencies (FastAPI has built-in WebSocket support)

## Files to Create

### `src/realtime/__init__.py`
Empty package init.

### `src/realtime/manager.py`
In-memory WebSocket connection manager.

- `ConnectionManager` class with:
  - `_connections: dict[int, set[WebSocket]]` — maps `user_id` to active WebSocket connections
  - `async connect(user_id: int, websocket: WebSocket)` — accept and register connection
  - `async disconnect(user_id: int, websocket: WebSocket)` — remove connection from set
  - `async notify_user(user_id: int, event_type: str = "dashboard_update")` — send JSON `{"type": event_type}` to all connections for that user. Catch and remove dead connections on send failure.
- Global singleton: `connection_manager = ConnectionManager()`

### `src/realtime/websocket.py`
FastAPI WebSocket endpoint.

- `websocket_endpoint(websocket: WebSocket)` at path `/ws/updates/`
- **Authentication flow:**
  1. Read `sessionid` cookie from WebSocket headers (`websocket.cookies.get("sessionid")`)
  2. Use Django's `SessionStore` to load session data
  3. Extract `_auth_user_id` from session
  4. Look up User via `user_repository.get_by_id()`
  5. If any step fails → `websocket.close(code=4401)` (custom close code for auth failure)
- **Connection loop:**
  1. After auth, call `connection_manager.connect(user_id, websocket)`
  2. Enter receive loop: `await websocket.receive_text()` (keeps connection alive, handles client pings)
  3. On `WebSocketDisconnect`: call `connection_manager.disconnect(user_id, websocket)`
- **Keep-alive:** Server sends `{"type": "ping"}` every 30 seconds via a background asyncio task. Client responds with any message to confirm liveness.
- Create a `router = APIRouter()` and register the endpoint on it.

### `frontend/src/composables/useRealtimeSync.js`
Vue composable for WebSocket client.

- Determines WebSocket URL from `window.location` (`ws://` or `wss://` based on protocol)
- Path: `/ws/updates/`
- On open: connection established (no logging to user)
- On message: parse JSON, if `type === "dashboard_update"`, call `router.reload()` from `@inertiajs/vue3`
- On close/error: auto-reconnect with exponential backoff (1s → 2s → 4s → 8s → 16s cap), silent
- On component unmount: close WebSocket, cancel reconnect timer
- Expose: `{ isConnected }` ref for optional debugging (not displayed in UI)

**Revised approach:** Only add notifications in `habit_service.py` (not dashboard views). The service is the single point where all completion/revert operations go through.

## Algorithm: WebSocket Authentication

1. Browser opens WebSocket to `/ws/updates/`
2. Browser automatically sends cookies (including `sessionid`) with the upgrade request
3. Server reads `sessionid` cookie from `websocket.cookies`
4. Server loads Django session via `importlib.import_module(settings.SESSION_ENGINE).SessionStore(session_key)`
5. Server reads `_auth_user_id` from session data
6. Server queries `User` model to verify user exists and is active
7. If valid: accept connection, register in manager
8. If invalid: close with code 4401

## Algorithm: Notification Flow

1. Bot/API/Web calls `habit_service.process_habit_completion()`
2. Service completes all DB operations (atomic transaction, audit log)
3. Service calls `connection_manager.notify_user(user.id)`
4. Manager iterates over all WebSocket connections for that user_id
5. Sends `{"type": "dashboard_update"}` to each
6. Dead connections (send raises exception) are removed from the set
7. Frontend receives message, calls `router.reload()` which re-fetches current Inertia page props from server

## Algorithm: Auto-Reconnect

1. WebSocket closes (server restart, network drop, etc.)
2. Composable schedules reconnect after `delay` (starts at 1s)
3. On each failed attempt: double the delay (cap at 16s)
4. On successful reconnect: reset delay to 1s
5. On component unmount: cancel any pending reconnect timer, close WebSocket
